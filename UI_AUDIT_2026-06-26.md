# Fincept Terminal — UI Audit

**Date:** 2026-06-26
**Auditor:** Devin (automated, evidence-grounded)
**Scope:** `apps/dashboard` — Next.js 14 (App Router) · TypeScript · Tailwind · Radix · TanStack Query · Zustand · Recharts · Framer Motion
**Specs referenced:** `docs/DESIGN.md` (design system), `apps/dashboard/src/lib/design-tokens.ts` (semantic intent contract), `apps/dashboard/tailwind.config.ts`, `apps/dashboard/src/app/globals.css`
**Prior audits:** `aestheticaudit.md` (2026-05-09), `docs/ui-audit-2026-05-16.md`, `apps/dashboard/docs/ui-audit-2026-06-03.md`, `apps/dashboard/docs/ui-upgrade-2026-06-03.md`

---

## 1. Executive Summary

The dashboard has a **coherent Bloomberg-terminal shell** (TitleBar → SafetyStateBar → NavTabs → main canvas → StatusBar), strong monospace typography, tabular figures, and a well-thought **semantic-intent color contract** (`design-tokens.ts`). The live-data grammar (pulse-on-update, LED dots, READY/WATCH/DEGRADED language) is genuinely good.

However, the implementation has **drifted from its own design system** in ways that are silently breaking the UI today, not just aesthetic nits:

1. **Broken color tokens.** `design-tokens.ts` emits `text-amber` / `bg-amber` / `border-amber` for the `degraded` intent, but `amber` is **not defined** anywhere in `tailwind.config.ts` or `globals.css`. These 13 usages produce **no styles** — every "degraded/stale/caveat" surface is uncolored. This is a live defect, not a polish item.
2. **Two conflicting design directions.** `globals.css` implements a flat "Bloomberg" aesthetic (`--radius: 0px`, zero corners, flat borders, ALL-CAPS), while `docs/DESIGN.md` specifies a TE/Nothing-OS aesthetic (cobalt focus, glass panels, neumorphic inset/raised, glow, `min-height: 100dvh`). The codebase is split between the two and commits to neither.
3. **Radius anarchy.** `--radius: 0px` is set globally, but `rounded-md/lg/xl/sm/full` appears **108 times** across the source. Inputs, dialogs, buttons, cards, pills, and the login hero all render rounded while the shell renders square — a visible mismatch on every screen.
4. **Dead "canonical" components.** `WidgetFrame` (declared the canonical container that "replaces Card") is used in **0 pages**. `Sidebar` is used in **0 pages**. 47 files still use `Card`. The codebase carries two abandoned navigation systems and two abandoned container systems.
5. **Accessibility gaps vs. spec.** `prefers-reduced-motion` is handled **0 times** despite infinite LED/pulse animations and a DESIGN.md mandate. Focus rings are inconsistent (present on `Button`, absent on most ad-hoc `<button>`s in pages). Nav links have no visible focus-visible state.
6. **Primary/accent mismatch.** `--primary` is orange (`28 100% 52%`), so active nav, primary buttons, and focus rings are orange. `DESIGN.md` §2 explicitly reserves cobalt (`#2F6BFF`) for "focus, active navigation, and measuring light" and orange for "paper-mode, warnings." The brand accent is wired backwards.
7. **Navigation divergence.** `nav-tabs.tsx` and `sidebar.tsx` define **different route sets**. Nav-tabs omits `/backtest`; sidebar omits `/reconciliation` and `/research`. Users can't reach all routes from either nav.

**Bottom line:** the shell and data grammar are production-grade; the token layer, radius discipline, dead components, a11y, and nav consistency are not. Items 1, 5, 6, and 7 should be fixed before the next release because they are functional defects, not styling preferences.

---

## 2. Methodology

- Static read of every shell component, every UI primitive (`components/ui/*`), the global stylesheet, the Tailwind config, the design-token module, and representative pages (`/`, `/login`, `/positions`).
- Quantified inconsistencies with project-wide ripgrep counts: `amber` token usage, `rounded-*` usage, `prefers-reduced-motion` presence, `WidgetFrame` vs `Card` adoption, `<Sidebar>` usage, viewport-height primitives.
- Cross-checked implementation against the two authoritative specs: `docs/DESIGN.md` and `apps/dashboard/src/lib/design-tokens.ts`.
- Findings are severity-tagged: **P0 (broken/functional)**, **P1 (spec violation, visible)**, **P2 (consistency/polish)**.

## 3. Findings

### 3.1 P0 — `amber` color token is undefined; `degraded` intent is silently broken

**Evidence**

- `apps/dashboard/src/lib/design-tokens.ts:37,46,55,64` emit `text-amber`, `bg-amber/10`, `border-amber/40`, `bg-amber` for the `degraded` SemanticIntent.
- `apps/dashboard/tailwind.config.ts` defines `long`, `short`, `warn`, `cyan`, `info`, `cobalt`, `hairline` — **no `amber`**. Default Tailwind requires a shade (e.g. `amber-400`); bare `text-amber` is not a valid class.
- `grep "amber" tailwind.config.ts` → 0 matches.
- `grep "text-amber|bg-amber|border-amber" src` → **13 matches** across `design-tokens.ts` and consumers.

**Impact**
Every surface that derives color from `healthIntent(false, true)`, `freshnessIntent(stale)`, `severityIntent("warning")`, or `sourceIntent` for degraded states renders with **no text/background/border color** — falling back to inherited foreground. Stale data, partial services, and warnings lose their visual signal. This directly undermines the product's core "verified vs degraded vs critical" grammar and the P0 guardrail in the safety bar.

**Fix**
Add `amber` to the Tailwind config mapped to a CSS var, and define `--amber` in `globals.css` (the existing `--warn: 42 100% 52%` is already amber-hued; either alias `amber` → `warn` or introduce a distinct `--amber`). Then either:

- keep `degraded` mapped to `amber` (preferred, distinct from `warn` badge), or
- remap `degraded` to the existing `warn` token and update the 13 call sites.

Either way, add a `design-tokens` test that asserts every `INTENT_*` class resolves to a defined Tailwind color.

### 3.2 P1 — Two conflicting design languages; implementation commits to neither

**Evidence**

- `globals.css:5-10` comment: *"Bloomberg / Fincept terminal aesthetic. Near-black surfaces, bright terminal semantics, zero corner radius, monospace, ALL-CAPS."* Sets `--radius: 0px` (`globals.css:33`), flat `1px solid` borders, no glass, no neumorphic shadows.
- `docs/DESIGN.md:5` describes *"quiet OLED research instrument … glass panes, soft inset pressure, LED state dots, cobalt/orange measurement marks … Teenage Engineering … Nothing OS."* Specifies glass panels (§7), neumorphic inset/raised shadows (§7), glow on active regions (§7), rounded instrument cards, and `min-height: 100dvh` (§4).
- The 2026-06-03 audit already flagged this as "Visual identity mismatch" (cobalt+orange on glass/OLED wanted; cyan-dominated flat borders shipped). It is still unresolved.

**Impact**
The product looks like a flat Bloomberg terminal while the spec markets a tactile glass instrument. Contributors cannot tell which direction is canonical, so new code drifts further. The `te-ui-design-savant` skill exists in the environment and is explicitly tuned for the TE/Nothing direction `DESIGN.md` describes — it is currently unused here.

**Fix**
Pick one direction and make the token layer enforce it. Recommendation: honor `DESIGN.md` (it is the newer, richer spec) and migrate `globals.css` to the cobalt/glass/neumorphic token set, OR formally retire `DESIGN.md` and update it to describe the flat Bloomberg aesthetic actually shipped. Do not leave both authoritative.

### 3.3 P1 — Radius anarchy (108 `rounded-*` usages vs `--radius: 0px`)

**Evidence**

- `globals.css:33` → `--radius: 0px`; `tailwind.config.ts:69-73` maps `lg/md/sm` all to `var(--radius)` (i.e. 0px).
- `grep "rounded-(md|lg|xl|sm|full)" src` → **108 matches**.
- Hotspots:
  - `components/ui/input.tsx:14` — `rounded-md` + `shadow-sm` (Input is rounded + shadowed while everything else is square/flat).
  - `components/ui/dialog.tsx:38` — `sm:rounded-xl` on `DialogContent`.
  - `components/ui/dialog.tsx:44` — `rounded-sm` on close button.
  - `app/login/page.tsx:59,109,136` — `rounded-lg`, `rounded` (code block), `rounded-md` (error box).
  - `components/shell/sidebar.tsx:56,80,93` — `rounded-md` on logo, nav links, kbd.
  - `components/shell/topbar.tsx:34,53,79,107` — `rounded-md` on every pill.
  - `app/positions/page.tsx:124,130` — `rounded-md` on "Show flat"/"Refresh" buttons.
  - `components/widgets/kpi-tile.tsx:47` — `rounded-md` on icon chip.

**Impact**
On any given screen the shell strips are square while the content cards/inputs/dialogs are rounded. The mismatch is immediately visible. It also makes the `--radius` token meaningless — authors bypass it with literal `rounded-*` classes.

**Fix**

- Decide the radius story (see 3.2). If flat-terminal: remove all literal `rounded-*` and let `--radius` govern; add an ESLint/Tailwind preset or a `design-tokens` test that forbids literal `rounded-*` outside allowlisted components (e.g. `rounded-full` for status dots).
- If instrument-glass: set `--radius` to the spec value and replace literal classes with `rounded-lg`/`rounded-md` consistently.
- Either way, `Input` and `Dialog` must stop hardcoding a radius that disagrees with the rest of the system.

### 3.4 P1 — `--primary` is orange; spec reserves cobalt for focus/active nav

**Evidence**

- `globals.css:20` → `--primary: 28 100% 52%` (orange `#FF7A1A`-ish); `globals.css:32` → `--ring: 28 100% 52%` (orange focus ring).
- `components/shell/nav-tabs.tsx:62-63,78` — active tab uses `bg-primary/10 text-primary` + `bg-primary` underline → **orange active nav**.
- `components/ui/button.tsx:15` — primary button is orange.
- `docs/DESIGN.md:31` — *"Cobalt is the tertiary accent and should appear as focus, active navigation, and measuring light."* `docs/DESIGN.md:32` — *"Orange is a complementary balancing color for paper-mode, warnings."*
- `globals.css:43-44` defines `--cobalt`/`--cobalt-soft` but they are not wired to `primary` or `ring`.

**Impact**
Active navigation and primary CTAs read as "warning/paper" orange, competing with the genuine paper-mode/warn semantics in the safety bar and title bar (`text-warn` PAPER chips). Focus rings are orange instead of cobalt, so the "measuring light" identity is lost. Brand identity is inverted relative to spec.

**Fix**
Either (a) set `--primary` and `--ring` to `--cobalt` (`220 100% 60%`) and reserve orange for paper/warn/destructive-adjacent emphasis, or (b) update `DESIGN.md` to bless orange as the primary. Given the safety bar already uses orange for PAPER/warn, option (a) is the cleaner reconciliation.

### 3.5 P1 — `prefers-reduced-motion` is not respected (0 matches)

**Evidence**

- `grep "prefers-reduced-motion" src` → **0 matches**.
- Infinite/persistent animations: `globals.css:145-170` (`pulse-update`, `animate-pulse-slow`, `live-dot`); `tailwind.config.ts:82-87` (`pulse-slow`, `spin-slow`, `ticker`); plus Framer Motion entrances in `kpi-tile.tsx:38-42` and `page.tsx` (motion wrappers).
- `docs/DESIGN.md:147` — *"Respect `prefers-reduced-motion`; persistent pulsing LEDs must become static."*

**Impact**
Operators sensitive to motion (and screenshot/OCR workflows) cannot suppress pulsing LEDs, row pulses, or ticker animations. This is an a11y violation of the project's own spec and a WCAG 2.1 SC 2.3.3 concern.

**Fix**
Add a global `@media (prefers-reduced-motion: reduce)` block in `globals.css` that disables `animation`/`transition` on `.live-dot`, `.animate-pulse-slow`, `.pulse-update`, `.animate-spin-slow`, `.animate-ticker`, and gate Framer Motion `animate` props on a `useReducedMotion()` hook (already shipped with framer-motion).

### 3.6 P1 — Navigation divergence between `nav-tabs` and `sidebar`

**Evidence**

- `components/shell/nav-tabs.tsx:25-41` (15 items): OVERVIEW, POSITIONS, ORDERS, RECON, STRATEGIES, OPTIMIZER, NEWS, RESEARCH, NEWS LAB, PREDICTIONS, MARKETS, MODELS, RECEIPTS, RISK, SYSTEM. **No `/backtest`.**
- `components/shell/sidebar.tsx:34-49` (14 items): Overview, Positions, Orders, Strategies, Optimizer, News, News Lab, Predictions, Markets, Backtest, Models, Receipts, Risk, System. **No `/reconciliation`, no `/research`.**
- `app-shell.tsx:48-52` renders `NavTabs` (horizontal) — `Sidebar` is **not mounted anywhere** (`grep "<Sidebar" src` → 0 files).

**Impact**
`/backtest` is unreachable from the active nav. `Sidebar` is dead code that still diverges from the live nav, so any future switch to a sidebar would silently drop Reconciliation and Research. The command palette (`command-registry.ts`) likely has its own third list — the three should be a single source of truth.

**Fix**

- Delete `sidebar.tsx` (dead) or restore it as the single nav and delete `nav-tabs.tsx`; do not keep both.
- Extract one `NAV_ITEMS` array (with route, label, mnemonic, icon) and have `nav-tabs`, the command registry, and any sidebar consume it. Ensure `/backtest`, `/reconciliation`, `/research`, `/watchlist`, `/quant-foundry/*` are all represented consistently.

### 3.7 P2 — `WidgetFrame` is the declared canonical container but is unused; `Card` is used in 47 files

**Evidence**

- `components/widgets/widget-frame.tsx:6-10` — *"This is the canonical content container going forward — replaces `<Card>`/`<CardHeader>`/`<CardTitle>` for the Bloomberg aesthetic."*
- `grep "WidgetFrame" src` → only the definition file (3 matches, all self-references). **0 consumers.**
- `grep "from \"@/components/ui/card\"" src` → **47 files**.

**Impact**
The "canonical" container is dead. The team maintains two container APIs (`Card` + `WidgetFrame`) and only one is real. New authors following the docstring will build against an untested component.

**Fix**
Either (a) execute the migration: adopt `WidgetFrame` across the 47 consumers and delete the `Card` header/title styling, or (b) delete `WidgetFrame` and update its docstring to reflect that `Card` remains canonical. Do not ship a "canonical" component with zero consumers.

### 3.8 P2 — Type scale and spacing tokens are not implemented

**Evidence**

- `docs/DESIGN.md:40-50` defines a full type scale (Display 48, H1 32, H2 24, H3 18, Body 14, Caption 11, Overline 10) and `docs/DESIGN.md:68-80` defines a 4px-based spacing scale (`--space-1`…`--space-12`).
- `tailwind.config.ts:78-81` only extends fontSize with `2xs`/`xxs`; no Display/H1/H2/H3 tokens. Pages use ad-hoc `text-2xl` (`page-header.tsx:14`), `text-3xl` (`kpi-tile.tsx:56`, `login/page.tsx:71`), `text-xl` (`login/page.tsx:63`) — `PageHeader` h1 is 24px, spec says H1 = 32px.
- No `--space-*` CSS variables in `globals.css`; `tailwind.config.ts` doesn't extend `spacing`. Pages use ad-hoc `p-2/p-3/p-5/px-4/gap-4`. `widget-body` uses `6px 8px` (`globals.css:94-96`) — 6px is off the 4px grid.

**Impact**
Headers are smaller than spec; spacing is inconsistent and occasionally off-grid. There is no single knob to tune the type/spacing rhythm.

**Fix**
Implement the spec tokens: add `--space-1..12` to `globals.css`, extend `tailwind.config.ts` `spacing` and `fontSize` (e.g. `display`, `h1`, `h2`, `h3`, `overline`), and migrate `PageHeader`/`KpiTile`/login hero to them. Add a `design-tokens` test asserting the scale exists.

### 3.9 P2 — Viewport height uses `h-screen`/`min-h-screen`, not `100dvh`

**Evidence**

- `grep "h-screen|min-h-screen|100dvh|min-h-dvh" src` → 5 matches, all `h-screen` or `min-h-screen`; **0 `dvh`**.
- `app-shell.tsx:46` → `flex h-screen flex-col overflow-hidden`; `layout.tsx:30` → `min-h-screen`; `login/page.tsx:53-54` → `min-h-screen`.
- `docs/DESIGN.md:90` — *"Prefer `min-height: 100dvh` over fixed viewport height."*

**Impact**
On mobile browsers `100vh` includes the URL bar and can clip the status bar / cause overscroll bounce. `100dvh` tracks the dynamic viewport. Also `h-screen` + `overflow-hidden` on the shell prevents the body from expanding in print (already patched with `print:h-auto`, but the base primitive is still `vh`).

**Fix**
Swap `h-screen` → `h-dvh` (or `min-h-dvh` where appropriate) in `app-shell.tsx`, `layout.tsx`, and `login/page.tsx`. Keep the `print:*` overrides.

### 3.10 P2 — Inconsistent focus-visible treatment; ad-hoc buttons lack focus rings

**Evidence**

- `components/ui/button.tsx:10` defines `focus-visible:ring-2 focus-visible:ring-ring` — good.
- Ad-hoc `<button>`s in pages bypass `Button`: `positions/page.tsx:122-133` ("Show flat", "Refresh") use `rounded-md border …` with **no `focus-visible:ring`**. `title-bar.tsx:177-183` (Kill) and `:313-320` (Logout) similarly lack focus rings. `nav-tabs.tsx:56-81` links and `sidebar.tsx:77-96` links have no `focus-visible` style.
- `docs/DESIGN.md:101,120` — *"visible focus rings"* and *"visible focus ring and native disabled state."*

**Impact**
Keyboard users cannot see which nav link or ad-hoc button is focused. Inconsistent with the `Button` primitive and the spec.

**Fix**
Route all buttons through `<Button>` (or add a shared `.focus-ring` utility). Add `focus-visible:ring-2 focus-visible:ring-ring` to nav links and ad-hoc controls. Consider a global `:focus-visible` outline in `globals.css` as a safety net.

### 3.11 P2 — `SafetyStateBar` and `StatusBar` are not semantic landmarks

**Evidence**

- `title-bar.tsx:282` uses `<header>` (good). `nav-tabs.tsx:48` uses `<nav>` (good). `status-bar.tsx:56` uses `<footer>` (good). `app-shell.tsx:53` uses `<main>` (good).
- `safety-state-bar.tsx:132` uses a bare `<div>` for a top-level strip between header and nav. `sidebar.tsx:54` uses `<aside>` (fine, but dead).

**Impact**
Screen readers get header/nav/main/footer but the safety strip is an anonymous region. Minor, but the spec calls out semantic landmarks (`DESIGN.md:101`).

**Fix**
Wrap `SafetyStateBar` in a `<section aria-label="Safety state">` (or `<div role="region" aria-label="…">`) so it is announced as a named region.

### 3.12 P2 — `purple-400` for AI intent is not in the design palette

**Evidence**

- `design-tokens.ts:39,48,58,67` map the `ai` intent to `text-purple-400` / `bg-purple-400/10` / `border-purple-400/40` / `bg-purple-400`. `purple-400` is a default Tailwind color so it *resolves* (unlike `amber`), but `docs/DESIGN.md` §2 palette has **no purple** — it lists cobalt, orange, green, red, cyan, info. The 2026-06-03 audit also names "purple = AI" as an existing token, so this is a spec-vs-implementation gap, not a runtime bug.

**Impact**
AI/model-generated content uses a color that isn't part of the documented brand palette. Reconciling the palette (3.2) will force a decision: promote purple into `DESIGN.md` or remap `ai` to an existing token (e.g. cobalt or info).

**Fix**
Decide and document. If purple stays, add `--ai` to `globals.css` + `tailwind.config.ts` and update `DESIGN.md` §2. If not, remap the `ai` intent and update consumers.

### 3.13 P2 — No light theme despite spec defining one; `dark` is hardcoded

**Evidence**

- `layout.tsx:27` → `<html className="dark …">` (always dark). `globals.css` defines only `:root` (dark values) + `@media print`; there is no `.light` / `:root.light` override. `tailwind.config.ts:5` → `darkMode: "class"`.
- `docs/DESIGN.md:11-27` defines both Light and Dark columns for every color.

**Impact**
Not a bug — the product is intentionally dark-first — but the spec carries a light palette that is unimplemented and `darkMode: "class"` is configured for a toggle that doesn't exist. Either ship the toggle or prune the spec's light column to avoid implying support.

**Fix**
Either implement a class-based light theme (alias the `:root` tokens to the Light column under `.light`) and add a toggle, or update `DESIGN.md` to mark the product dark-only and remove the Light column.

## 4. Prioritized Remediation Plan

### Phase 0 — Functional defects (do first, small, safe)

1. **Define `amber`** (or remap `degraded` → `warn`) and add a `design-tokens` test asserting all `INTENT_*` classes resolve. *(3.1)*
2. **Reconcile navigation** to a single `NAV_ITEMS` source; delete the dead `sidebar.tsx` or restore it; ensure `/backtest`, `/reconciliation`, `/research`, `/watchlist`, `/quant-foundry` are all reachable. *(3.6)*
3. **Add `prefers-reduced-motion` handling** in `globals.css` + Framer `useReducedMotion()`. *(3.5)*
4. **Add focus-visible rings** to nav links and ad-hoc buttons (or route them through `<Button>`). *(3.10)*

### Phase 1 — Token & identity reconciliation

5. **Pick one design direction** (flat Bloomberg vs TE/Nothing glass) and update the loser spec. *(3.2)*
6. **Fix `--primary`/`--ring`** to cobalt (or bless orange in spec). *(3.4)*
7. **Resolve radius discipline** — enforce `--radius` and remove/allowlist literal `rounded-*`. *(3.3)*
8. **Decide purple fate** — promote to spec or remap `ai` intent. *(3.12)*

### Phase 2 — Component hygiene

9. **Execute or retire `WidgetFrame`** — either migrate the 47 `Card` consumers or delete `WidgetFrame` and fix its docstring. *(3.7)*
10. **Implement type + spacing tokens** from `DESIGN.md` §3-4 and migrate `PageHeader`/`KpiTile`/login. *(3.8)*
11. **Switch to `100dvh`**. *(3.9)*
12. **Semantic-ize `SafetyStateBar`** as a labelled region. *(3.11)*

### Phase 3 — Optional

13. Light theme: ship the toggle or prune the spec's Light column. *(3.13)*

## 5. What is already good (do not regress)

- **Shell architecture** (`app-shell.tsx`): TitleBar → SafetyStateBar → NavTabs → main → StatusBar is clean, single-purpose, dense, and prints correctly via `print:*` overrides.
- **Semantic-intent contract** (`design-tokens.ts`): the *idea* of cyan=verified, amber=degraded, red=critical, purple=ai, green=healthy, gray=inactive is excellent — it just needs the `amber` token wired up and the palette reconciled.
- **Live-data grammar**: `pulse-update` row flash, `live-dot` LED, READY/WATCH/DEGRADED safety language, freshness badges, mock badges — strong operator affordances.
- **Typography base**: monospace everywhere, `tnum`/`zero` font features, tabular-nums on tables — exactly right for a trading terminal.
- **Print stylesheet** (`globals.css:173-291`): a genuinely thoughtful `@media print` block with ink-save, page-break-avoid, and report-page rules.
- **Query/auth hygiene** (`providers.tsx`): sane react-query defaults, 401 no-retry, JWT hydration guard.
- **Test scaffolding**: per-component test scripts (`package.json` scripts) for design tokens, LED dot, mock badge, signal card, trading chart, watchlist row, etc.
