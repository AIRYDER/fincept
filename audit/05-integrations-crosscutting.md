**Scope:** `mcps/`, `experiments/`, `research/`, `strategies/`, top-level config/docs/data/reports, and cross-cutting concerns (security, testing, deployment, CI/CD, dependencies, observability).
**Method:** Read-only audit of every relevant file. No source modified.
**Date:** 2026-06-26

## Executive Summary

The audited surface is a mix of **high-quality, deliberately-isolated research artifacts** (`experiments/news-impact-model`, `research/`) sitting alongside a **large, partially-stale documentation sprawl** and a **local-only MCP tool catalog** that is not part of the committed product.

Five bullets capture the state:

1. **`mcps/` is a gitignored local artifact, not a product surface.** `.gitignore:119` ignores `/mcps/`. Each subdirectory is just a `tools/*.json` schema catalog regenerated by external MCP servers (BridgeMemory, BridgeMind, Chrome DevTools, Cloudflare, Exa, Firecrawl, GitHub, Notion, Vercel, Neon). There is no server code, no manifest, no wiring into the Fincept runtime — they are reference schemas for AI-agent tool calls. **One file leaks a Cloudflare account ID and operator email** (`mcps/cloudflare-api/tools/execute.json:3` hardcodes `accountId = "5a3014166be2e7c87846feeb272386c3"` and `Nolans999@gmail.com`). Although gitignored, this is a local-secrets-hygiene defect.

2. **`experiments/news-impact-model` is the best-engineered research artifact in the repo.** Clean contract-first design (frozen dataclasses, point-in-time `available_at_ns`), deterministic analog retrieval, leave-one-out + walk-forward weight optimization, a standalone HTTP workbench, and 27 focused tests. **One test is currently failing** (`test_label_event_impact_supports_beta_adjusted_abnormal_returns` calls `label_event_impact(..., asset_beta=1.5)` but `labels.py` has no `asset_beta` parameter), and the `IMPLEMENTATION_STATUS.md` claim of "14 passed" is stale (actual: 26 passed, 1 failed). The experiment is correctly isolated from the uv workspace and main service graph.

3. **`research/` is a well-governed curated reference database** — 51 entries across papers/repos/models/architectures/benchmarks/events, with a strict frontmatter schema, tier-to-entry roadmap mapping, anti-curation policy, and update log. This is a genuine asset. The only weakness is that every entry shares `last_reviewed: 2026-06-22` (the seed date), so the "review every 6/12 months" maintenance policy has never actually fired.

4. **`strategies/` is nearly empty** — just `alpaca.live.json` (one `position_tracker` strategy over 11 symbols) and its 2-line history JSONL. There is no strategy framework, no registry, no validation, and no documentation of how strategies are defined or run from this directory (the real strategy_host service lives under `services/strategy_host`). The directory is a runtime data dump, not a designed surface.

5. **Top-level config and CI/CD are strong; documentation is bloated and drifting.** `pyproject.toml`, `ruff.toml`, `.pre-commit-config.yaml`, `.gitleaks.toml`, and the four SHA-pinned GitHub workflows are professional-grade. But the root and `docs/` contain **~60 markdown files (>2MB)** with heavy overlap: at least 8 "next steps / what next / big plan / consolidated report" documents, multiple stale progression snapshots embedded directly in `README.md`, and several CI-breaking drifts already flagged in `docs/audits/2026-06-25/AUDIT_2026-06-25.md` (dead `build-images.yml` Dockerfile references, hallucinated LLM model names, env-var catalog mismatches). The documentation load is itself a maintenance risk.

## MCP Server Integrations

### Overview

`mcps/` contains 11 subdirectories, each with a single `tools/` folder of JSON files. **There is no Python/JS server code, no `pyproject.toml`, no `package.json`, no manifest, no README, and no config.** Each JSON file is a standalone MCP tool definition: `name`, `description`, and a JSON-Schema `inputSchema`. These are the schemas external MCP servers expose to AI agents; they are not Fincept code.

The entire `mcps/` tree is **gitignored** (`.gitignore:119` — `/mcps/`), labeled "Local MCP server configs (regenerated by MCP servers)". So this directory is local working state, not a committed product surface. It should not be treated as part of the system's auditable contract.

### What's optimal

- The schemas are well-formed JSON-Schema (draft-07 or 2020-12) with `additionalProperties: false` on most, proper `required` arrays, and rich `description` fields that guide agent usage.
- Destructive tools (Neon `run_sql`, `run_sql_transaction`) carry explicit "ask the user first" guidance in their descriptions — good prompt-level safety.
- The directory is gitignored, so it does not pollute the committed repo.

### What needs work

- **CRITICAL (local hygiene):** `mcps/cloudflare-api/tools/execute.json:3` embeds a real Cloudflare account ID and the operator's personal email in the tool description. Even though gitignored, this file is generated by the Cloudflare MCP server and should be sanitized or regenerated without PII. Anyone with filesystem access (or a screen-share) sees the account ID.
- **No manifest or README:** A new contributor has no way to know what these directories are, which MCP servers generate them, or whether they're committed. A single `mcps/README.md` (gitignored alongside, or a committed note in `docs/`) explaining "these are regenerated local MCP tool catalogs, do not edit" would prevent confusion.
- **No version pinning:** The tool schemas can drift as MCP servers update. There is no snapshot of which server version produced which schema, making agent-behavior reproducibility impossible.
- **Stale-schema risk:** Since the files are regenerated, an agent reading a stale `tools/*.json` may call a tool with a schema the server no longer supports. There is no freshness check.

### Better approaches

- Treat `mcps/` as pure cache: add a `mcps/.generated-by` marker file (gitignored) recording the MCP server name + version + generation timestamp, and have a cleanup script wipe `mcps/` on MCP server restart.
- Move the Cloudflare account ID out of the tool description entirely — the `account_id` parameter exists for exactly this reason; the hardcoded default in the description is a server-side misconfiguration to report upstream.
- If any of these MCPs are actually used by Fincept agents at runtime (BridgeMind for task delegation, Exa/Firecrawl for research), document the integration in `docs/` and add a contract test that the tool schemas Fincept relies on remain present.

## Experiments: news-impact-model

### Purpose

Isolated scaffold for a raw predictive model: **news item → predicted market impact (abnormal return distribution by symbol and horizon)**. Explicitly does **not** size trades, submit orders, or wire into the live Fincept runtime. It is intentionally outside the uv workspace and main service graph (`README.md:10-11`).

### Architecture

```
src/news_impact_model/
├── schema.py        # frozen dataclasses: NewsEvent, MarketContext, HistoricalOutcome,
│                    #   PricePoint, ImpactLabels, AnalogMatch, HorizonImpact,
│                    #   SimilarEventSummary, NewsImpactPrediction
├── labels.py        # label_event_impact: abnormal-return labels from asset/benchmark prices
├── data.py          # load_historical_outcomes (JSONL/JSON/CSV), write_historical_outcomes_jsonl
├── events.py        # vendor export normalization: NormalizedNewsEvent, EntityLinker,
│                    #   classify_event_type, dedupe_news_events, source_latency_stats
├── text.py          # tokenize, weighted_jaccard (deterministic text similarity)
├── analogs.py       # HistoricalAnalogIndex + AnalogScoringWeights (text/symbol/event/regime/source/recency)
├── features.py      # HashingTextEmbedder, VectorAnalogIndex, encode_market_context, event_surprise_features
├── model.py         # NewsImpactModel: analog retrieval → multi-horizon impact distribution
├── training.py      # evaluate/optimize_analog_weights (leave-one-out + walk-forward), source-event priors
├── evaluation.py    # purged_walk_forward_splits, holdout splits, calibration_curve,
│                    #   impact_decay_accuracy, error_analysis_by_event_source
├── pipeline.py      # NewsImpactPipeline facade
└── workbench.py     # WorkbenchState + ThreadingHTTPServer (local browser workbench)
```

Plus `scripts/` (demo, optimize_weights, serve_workbench), `tests/` (6 test files), `workbench/` (static HTML/CSS/JS), `sample_data/historical_outcomes.jsonl` (8 sample events), and `docs/` (4 markdown files).

### ML approach

The model is a **deterministic analog-retrieval baseline**, deliberately shaped like the production model but with no learned weights yet:

1. **Retrieval:** `HistoricalAnalogIndex.search()` scores each historical outcome against a new event using a weighted sum of text overlap (weighted Jaccard), symbol match, event-type match, regime match, source credibility, and recency (exponential decay, τ=730 days).
2. **Prediction:** `NewsImpactModel.predict()` retrieves top-k analogs and computes per-horizon `expected_return` (weighted mean), `p_up` (up-weight fraction), `q10/q50/q90` (quantiles), `volatility_impact`, `volume_impact`, and a heuristic `confidence` (top-score + coverage + text-quality blend, clamped to 0.05–0.98).
3. **Optimization:** `optimize_analog_weights` picks the lowest-MAE candidate from a small hand-crafted 5-point weight grid (`_default_weight_grid`), evaluated via leave-one-out or walk-forward (chronological, only-prior-events). Walk-forward records `train_event_ids` per fold for leakage auditing.
4. **Feature scaffolding:** `HashingTextEmbedder` (BLAKE2b hashing trick, 256-dim, L2-normalized) + `VectorAnalogIndex` provide a vector-retrieval contract for when real embeddings land. `encode_market_context` and `event_surprise_features` produce tabular features for future GBM/transformer heads.
5. **Evaluation toolkit:** `purged_walk_forward_splits` (with no-leakage gap), `event_type_holdout_split`, `source_holdout_split`, `calibration_curve` (bucketed p_up vs realized direction), `impact_decay_accuracy` (magnitude + sign + decay-shape), `error_analysis_by_event_source`.

### Data flow

```
vendor export (JSONL/JSON/CSV)
  → normalize_vendor_event (point-in-time available_at_ns, entity linking, event classification, dedupe)
  → HistoricalOutcome (with abnormal_returns by horizon)
  → HistoricalAnalogIndex (in-memory)
  → NewsImpactModel.predict(NewsEvent, MarketContext)
  → NewsImpactPrediction (per-horizon distribution + similar_events + confidence)
```

Labels are built by `label_event_impact` from asset + benchmark price series: base price = last observation at or before event time; horizon price = first observation at or after `available_at + h`; abnormal return = asset return − benchmark return.

### Test coverage

6 test files, **27 tests, 26 passing, 1 failing**:

| File | Tests | Coverage |
|---|---|---|
| `test_news_impact_model.py` | 3 | Analog retrieval ranking, label computation, multi-horizon prediction shape |
| `test_training_optimizer.py` | 4 | Leave-one-out exclusion, optimizer selection, walk-forward prior-only constraint, walk-forward candidate selection |
| `test_workbench.py` | 3 | Dataset loading/profile, optimization, prediction with optimized weights |
| `test_data_io.py` | 4 | JSONL/CSV loading, round-trip, `optimize_weights.py` subprocess script |
| `test_news_data_foundation.py` | 7 | Vendor normalization, entity linker, event classifier, dedupe, latency stats, **beta-adjusted labeling (FAILING)** |
| `test_model_evaluation_foundation.py` | 6 | Hashing embedder, vector index, context/surprise features, purged walk-forward, holdout splits, calibration/decay/error |

**Failing test:** `test_label_event_impact_supports_beta_adjusted_abnormal_returns` (`tests/test_news_data_foundation.py:158-173`) calls `label_event_impact(..., asset_beta=1.5)`, but `labels.py::label_event_impact` has signature `(event_available_at_ns, asset_prices, benchmark_prices, horizons_ns)` — no `asset_beta` parameter. This is a real defect: either the test is ahead of the implementation (beta-adjustment was planned but not landed) or the test was added without running it. The `IMPLEMENTATION_STATUS.md` claim of "14 passed" is stale — the test suite has grown to 27 tests and one is red.

Tests use `sys.path.insert(0, str(ROOT / "src"))` to import without installing the package — appropriate for an out-of-workspace experiment.

### What's optimal

- **Contract-first design.** Frozen dataclasses with `extra="forbid"` semantics via `@dataclass(frozen=True)`, strict point-in-time `available_at_ns` everywhere, and a clean `NewsImpactPrediction` output contract that the downstream system can consume without knowing the internals.
- **Correct leakage discipline.** Walk-forward optimization sorts by `available_at_ns`, only uses prior events, and records `train_event_ids` per fold. `purged_walk_forward_splits` adds a no-leakage gap. This is the right quant discipline, not just "split randomly".
- **Deterministic and dependency-free.** No numpy/pandas/sklearn/torch needed for the baseline — pure stdlib. Makes the scaffold testable and fast.
- **Honest about limitations.** `README.md:158-169` and `IMPLEMENTATION_STATUS.md:155-198` explicitly list what's missing (no embeddings, no vector DB, no trained text transformer, no calibration, no Fincept wiring). The promotion path is staged in 4 slices with guardrails ("Never submit orders from this agent", "Never emit a signal without `available_at_ns`").
- **Workbench is research-only.** Local HTTP server on 127.0.0.1:8765, no order routing, no signal publishing. Path-traversal protection on static file serving (`workbench.py:262`).

### What needs work

- **1 failing test** (`asset_beta` not implemented). Either implement beta-adjusted abnormal returns in `labels.py` or remove/xfail the test. The test name promises a feature that doesn't exist.
- **Stale doc.** `IMPLEMENTATION_STATUS.md:233` says "14 passed" — actual is 26 passed + 1 failed. The doc hasn't been updated as tests were added.
- **`_default_weight_grid` is only 5 candidates.** The optimizer picks the best of 5 hand-crafted weight sets. This is a placeholder; a real optimizer (grid search, Optuna) is explicitly deferred but should be flagged as the next step.
- **`_coerce_epoch_number_to_ns` has a bug** (`events.py:488-498`): the `> 1e14` and `> 1e11` branches both multiply by `NS_PER_MILLISECOND` — the `> 1e11` branch (seconds) should multiply by `NS_PER_SECOND`. This means a epoch-in-seconds timestamp (e.g., `1700000000`) would be treated as milliseconds and produce a timestamp 1000× too large. Tests don't catch this because they pass `available_at_ns` directly or use ISO strings.
- **In-memory only.** `HistoricalAnalogIndex` and `VectorAnalogIndex` hold everything in Python lists — O(n) scan per query. Fine for the scaffold, but the README's "production version should use a vector index" is a real scaling cliff at ~10k+ events.
- **`__pycache__` committed in experiments.** `experiments/news-impact-model/src/news_impact_model/__pycache__/` and `tests/__pycache__/` exist on disk. The root `.gitignore` covers `__pycache__/` globally so these won't be committed, but they should be cleaned.
- **No `pyproject.toml` for the experiment.** It's intentionally outside the uv workspace, but there's no `pyproject.toml` even for dev/pytest config. Tests rely on `sys.path` hacks. A minimal standalone `pyproject.toml` with `pytest` config would be cleaner.

### Better approaches

- Fix the `asset_beta` test immediately (implement or xfail). A red test in a "best-engineered research artifact" undermines the contract-first claim.
- Fix `_coerce_epoch_number_to_ns` second/millisecond ambiguity and add a unit test for each magnitude branch.
- Add a `pyproject.toml` to the experiment root so `pytest` picks it up without `sys.path.insert` and so the package is pip-installable in isolation.
- Replace the 5-candidate weight grid with a proper optimizer (scipy.optimize or Optuna) once the analog baseline is validated against real data — the current code is structured for this, it just needs the candidate generator swapped.
- Add a `make test-experiment` or `scripts/test-experiment.ps1` target so the experiment's 27 tests run in CI as a separate job (they currently aren't in `pyproject.toml`'s `testpaths = ["libs", "services"]`).

Every entry is a single markdown file with YAML frontmatter (title, authors, affiliation, source, date, added, last_reviewed, status, relevance, tier_mapping, tags, license, effort_to_apply, adoption_risk) and a body (TL;DR, Why we care, Key ideas, How to apply to Fincept, Caveats, Related entries, References).

### What's useful vs stale

**Useful:**
- `_meta/ROADMAP_MAPPING.md` is genuinely the highest-leverage file — it maps every Sisyphus tier gap (Q0.1–Q4.5) to specific research entries with relevance ranking. A engineer implementing "Tier Q2.4 Kelly-optimal sizing" can open this file and immediately find 5 relevant papers.
- The "How to apply to Fincept" sections are concrete and cite specific files (`gbm_predictor/train.py::main`, `news_alpha_predictor/evaluate.py::CandidateGatePolicy`). This is not generic literature review — it's actionable.
- The `effort_to_apply` (S/M/L/XL) and `adoption_risk` (low/medium/high) fields make the entries triageable.
- The anti-curation policy (`_meta/ANTI_CURATION.md`) prevents the database from becoming a link dump.

**Stale/at-risk:**
- **Every entry has `last_reviewed: 2026-06-22`** — the seed date. The README promises "high entries reviewed every 6 months; medium and low every 12 months." No review has ever fired. As of 2026-06-26 (4 days later) this is technically not yet stale, but the maintenance process has no enforcement mechanism (no cron, no CI check, no reminder).
- The `INDEX.md` header has a corrupted character: `# Index ? Master Table of Contents` — the `?` should be `—` (em-dash encoding issue, likely from a Windows codepage mismatch). Same in `ROADMAP_MAPPING.md:1` (`# Roadmap Mapping ? Sisyphus Tier Gaps...`).
- The README references `Sisyphus_Quant_ML_Deep_Dive.md` and `Sisyphus_Ultra_Report.md` as companion docs — both exist in `docs/` and are large (50KB, 41KB). The research DB is tightly coupled to these audit docs; if they drift, the tier mappings become meaningless.
- `UPDATE_LOG.md:26` has a timestamp with a `?` artifact: `2026-06-22 20:04:00 ? Phase 2 expansion complete` — another encoding issue.

### Better approaches

- Add a CI check (or pre-commit hook) that validates every entry's frontmatter against `ENTRY_TEMPLATE.md` and verifies `last_reviewed` is within the review window (6mo for high, 12mo for medium/low). Fail when an entry goes stale.
- Fix the em-dash encoding artifacts in `INDEX.md`, `ROADMAP_MAPPING.md`, and `UPDATE_LOG.md`.
- Add a `research/_meta/SOURCE_HASHES.md` or similar that records the last-known URL/status of each entry's `source` link so link-rot is detectable.
- Consider moving the research DB into a dedicated `research/` git submodule or separate repo if it grows beyond ~100 entries — it's already 51 files and tightly governed, and it doesn't need to share commit history with the trading code.

## Strategies

### Structure

`strategies/` contains exactly 2 files:

- `alpaca.live.json` — a single strategy config: `strategy_id: "alpaca.live"`, `class_name: "position_tracker"`, 11 symbols (AI, AMD, AMZN, ARKQ, BOTZ, GOOGL, MSFT, MU, NVDA, PATH, TSLA), `params: {}`, `model_binding: null`, `enabled: true`, timestamps.
- `alpaca.live.history.jsonl` — 2-line append-only history of the config (first line `enabled: false`, second line `enabled: true`).

### How strategies work

This directory is **not** the strategy framework — it's a **runtime data dump** written by `services/strategy_host` (the filesystem-backed strategy instance supervisor referenced in `PROJECT_OVERVIEW.md:44` and `SYSTEM_OVERVIEW.md:41`). The actual strategy definitions, loading, and lifecycle live in the strategy_host service. This directory just persists the active config and its history.

There is no README, no schema documentation, no validation, and no indication of what `class_name: "position_tracker"` maps to (presumably a strategy class registered in the strategy_host).

### What's optimal

- Append-only history JSONL is a reasonable audit pattern — you can see when the strategy was enabled.
- The config is minimal and human-readable.

### What needs work

- **No README or schema.** A new contributor seeing `strategies/alpaca.live.json` has no idea what `class_name` values are valid, what `model_binding` expects, or whether `params` is freeform. The schema lives in `services/strategy_host` source, not next to the data.
- **No validation that `class_name` exists.** A typo in `class_name` would silently fail at strategy_host load time.
- **`enabled: true` on a strategy named `alpaca.live` with `class_name: "position_tracker"`** is ambiguous — is this a live-trading strategy or a paper one? The name says "live" but the system is paper-only by default (`FINCEPT_TRADING_MODE=paper`). This could cause operator confusion.
- **Only one strategy exists.** The directory implies a multi-strategy system but only one is configured. Either more strategies should exist, or the directory should be documented as "the active strategy config, managed by strategy_host".

### Better approaches

- Add a `strategies/README.md` explaining: "This directory holds active strategy configs persisted by `services/strategy_host`. Do not edit directly — use the API (`POST /strategies`) or the dashboard. Schema: see `libs/fincept-core/src/fincept_core/strategy_config.py`."
- Add a JSON Schema file (`strategies/schema.json`) and validate configs against it on write.
- Rename `alpaca.live` to something less ambiguous (e.g., `alpaca.paper.position_tracker`) to avoid the live/paper confusion, or add a `trading_mode: paper` field to the config.

## Cross-Cutting Concerns

### Security & secrets management

**Strong:**
- `.env` is gitignored; `.env.example` uses placeholders only.
- `.pre-commit-config.yaml` includes `detect-private-key` and `gitleaks`.
- CI `security` job runs gitleaks with full history (`fetch-depth: 0`).
- Nightly `vuln-scan` runs Trivy (CRITICAL/HIGH, exit 1) and `pip-audit`.
- `railway-production.json` explicitly documents safety invariants: paper-only, no broker creds on Railway, all secrets as `${{secrets.NAME}}` placeholders.
- RunPod handlers carry no broker credentials and sign callbacks with HMAC (`runpod/quant-foundry-training/README.md:9-23`).
- `docs/security_best_practices_report.md` documents the JWT-no-expiry finding (H1) and path-traversal gaps.
- R-15 in `RISKS.md` documents provider-evidence redaction at the storage boundary.

**Weak:**
- **`mcps/cloudflare-api/tools/execute.json:3`** hardcodes a Cloudflare account ID and operator email (local only, gitignored, but still a hygiene defect).
- **`railway-production.json:52-53`** hardcodes RunPod endpoint IDs (`h2blqodcicxqyy`, `t31u1z426jy1ub`). These are infrastructure identifiers, not secrets, but they're duplicated across multiple files (per `docs/audits/2026-06-25/AUDIT_2026-06-25.md`).

- **RunPod handler dev-secret fallback** (`runpod/quant-foundry-training/handler.py:45`) silently uses `dev-callback-secret-DO-NOT-USE-IN-PROD` instead of failing closed.
- **`.gitleaks.toml` has no project-specific rules** for Cloudflare account IDs or RunPod endpoint IDs.
