# Quant ML Audit — Fincept Terminal

**Date:** 2026-06-03
**Author:** Quant ML Builder (agent audit pass)
**Scope:** `libs/fincept-core`, `libs/fincept-bus`, `libs/fincept-db`, `libs/fincept-tools`, `libs/fincept-sdk`, `services/{backtester,oms,risk,portfolio,orchestrator,agents,strategy_host}`, `experiments/news-impact-model`, `strategies/`, `scripts/`.
**Out of scope:** dashboard UI, `services/api`, ingestor venue adapters, OpenBB integration.

---

## Executive Summary

The Quant/ML spine is in good shape for the project stage. The architectural foundations are right: a typed event bus, frozen Pydantic schemas, shared portfolio math, PIT-safe analytics tools, a paper-only execution gate, a risk gate used by both backtester and live OMS, and a working ML lifecycle (train → walk-forward CV → promote → hot-reload → shadow → predict → JSONL log). The `paper_spine_replay.py` script already proves the full data → feature → signal → decision → risk → order → fill → portfolio chain end-to-end.

The most material problems are concentrated in three places:

1. **Risk gate is incomplete.** `MAX_DAILY_LOSS_USD=2000` is configured but never enforced. Kill switch is in-memory and resets on OMS restart. There is no per-strategy drawdown cap and no regime awareness. **For a dad-as-operator deployment, the daily-loss enforcement gap is the single most important thing to close first.**

2. **Performance metrics are shallow.** The backtester report computes Sharpe and max drawdown only — no Sortino, no Calmar, no profit factor, no win rate, no exposure %, no recovery time, no baseline comparison. Live portfolio has no public tool to compute these from current positions. The same Sharpe is also computed in two places (analytics tool, backtester report) with slightly different formulas — drift risk.

3. **ML explainability and settlement are scaffolded but not wired.** The prediction log records `direction` + `confidence` + `horizon` but no feature contributions, no regime context, no rationale text, no top-k similar historical predictions. There is no settlement/outcome store, so active vs shadow comparison is impossible. The ML improvement report (`docs/project-understanding/10-ml-improvement-report.md`) already flags these as P0; this audit re-prioritizes them with concrete code paths.

Everything below is incremental, paper-only, and reversible. **No live trading path is proposed in this audit.** Where live-trading risks exist (Alpaca adapter, kill-switch persistence), they are called out but not patched.

## 1. Quant/ML Audit

### 1.1 Architecture review

The vertical flow is exactly what the IMPLEMENTATION.md spec calls for:

```
md.* (ingestor) -> features.* (features) -> sig.* (agents) -> decisions.* (orchestrator)
   -> risk.approved.* (risk) -> ord.* (OMS) -> fill.* -> ord.positions (portfolio) -> API/dashboard
```

**Strengths:**
- Single shared `apply_fill_to_position` in `libs/fincept-core/src/fincept_core/portfolio.py` is used by both `services/backtester/engine.py` and the live portfolio service. No drift risk for the four fill cases (open more / close some / cross flat / cross flip).
- The same `risk.check_intent` runs in `services/backtester/engine.py:_gate_intent` and `services/oms/src/oms/main.py`. A backtest mirrors the live gate by construction.
- `Strategy` ABC in `libs/fincept-sdk/src/fincept_sdk/strategy.py` plus a `StrategyContext` Protocol is honored by both backtester and live runtime. A strategy class written once runs in both places.
- `PredictionLog` (`libs/fincept-core/src/fincept_core/prediction_log.py`) is an append-only JSONL per agent — `cat`-able, schema-stable, fixture-friendly.

**Weaknesses:**
- The kill switch state lives in process memory (`services/risk/src/risk/state.py:KillSwitchState`) with an explicit comment that "On OMS restart the flag resets to False." For a dad-as-operator paper-trading deployment that's a manageable trade-off, but it is a real risk surface and should be flagged in operator docs.
- The OMS has two routers (`sim` / `alpaca`) selected at startup. The Alpaca adapter is live-capable (paper by default, but `ALPACA_BASE_URL` is configurable). Any code path that flips `OMS_ROUTER=alpaca` AND `ALPACA_BASE_URL=https://api.alpaca.markets` is a live-trading gate. Currently there is no UI/CLI that does this in one step, but the affordance exists.
- The orchestrator's consensus (`services/orchestrator/src/orchestrator/consensus.py`) keeps the latest prediction per `(agent_id, symbol)` in memory; an orchestrator restart loses that view until new predictions arrive. Not a safety issue, but it means the "consensus at time T" cannot be replayed.

### 1.2 Schemas & contracts (`libs/fincept-core/src/fincept_core/schemas.py`)

Models are `frozen=True, extra="forbid"`, all carry `schema_version: int = 1`. Fields:

| Model | Notes |
|---|---|
| `MarketEvent` / `TradeEvent` / `BarEvent` / `BookDeltaEvent` / `BookSnapshotEvent` | Good. Venue, asset_class, ts_event, ts_recv separation. |
| `Prediction` | Has `direction`, `confidence`, `horizon_ns`, `magnitude`, `calibration_tag`. **Missing: `rationale`, `feature_contributions`, `regime`, `model_class` (e.g., "gbm", "news_alpha", "regime").** |
| `SentimentSignal` / `NewsImpactSignal` | Schema is correct but `calibration_tag` is the only explainability hook. |
| `Decision` | Has `rationale: str` (good), `source_signals: list[str]` (good — preserves lineage). **Missing: `confidence` (only `urgency`).** |
| `OrderIntent` | No `confidence` field. **Decision confidence is lost at the order boundary.** |
| `RiskCheckResult` | `approved`, `reduced_notional_usd` (note: this is reserved but the live gate never sets it), `reasons: list[str]`. Reason format is "free text" — see gap 1.10. |
| `Position` | Good. `realized_pnl`, `unrealized_pnl`, `quantity` (signed), `avg_cost`. |

**Bug / smell:** `Decision.urgency` is a free `float` with no bounded validator. A negative urgency is meaningless but accepted.

### 1.3 Indicators / analytics (`libs/fincept-tools/src/fincept_tools/analytics/tools.py`)

Tools registered: `compute_returns`, `compute_vol`, `compute_correlation`, `compute_vwap`, `compute_sharpe`, `compute_drawdown`. All PIT-safe (accept `end_ns`), all annualize correctly for 1m/1h/1d.

**Strengths:**
- PIT safety is real: every tool's `_run` reads from `[0, end_ns]` then slices the last N bars, never peeks.
- `compute_sharpe` handles a small `1e-12` tolerance for FP near-constant inputs — operator-friendly.

**Weaknesses:**
- **No `compute_sortino`**, **no `compute_calmar`**, **no `compute_beta`**, **no `compute_alpha`**, **no `compute_var` / `compute_cvar`**. The `compute_sharpe` docstring claims annualization, but Sortino and Calmar are the standard pair you'd want for a risk-aware trader.
- All tools operate on bars from a single symbol (or pair for correlation). **No tool aggregates across the live portfolio's positions** — see gap 1.6.
- VWAP fallback uses `(high+low+close)/3` when `vwap` is null, but this is undocumented. Operators reading a "vwap=210.45" cannot tell whether the bar's real VWAP was 210.45 or whether the bar had no vwap field and we computed a proxy. Add a `vwap_source: "bar" | "proxy"` field.

### 1.4 Strategies / strategy SDK

The Strategy SDK is small and well-designed. `StrategyContext` is a `Protocol` (structural typing). Strategies are loaded from a registry in `services/backtester/src/backtester/strategies.py` (referenced indirectly; not opened in this audit).

**Weakness:** `Strategy` declares class attributes `strategy_id` and `symbols` as `ClassVar[str]` / `ClassVar[list[str]]`, but does not declare `params: dict`. A new strategy's parameter shape is opaque to the host until the strategy is instantiated. The strategy host (`services/strategy_host/src/strategy_host/runner.py`) probably handles this — verify before adding new strategy classes.

**Concrete strategies present** (from `strategies/alpaca.live.json`): `position_tracker` with 11 symbols (AI, AMD, AMZN, ARKQ, BOTZ, GOOGL, MSFT, MU, NVDA, PATH, TSLA). `params: {}`, `model_binding: null`. This is a "track only" strategy — it doesn't emit orders; the host's role here is presumably to surface these symbols in the dashboard and let the user place manual orders.

**Strengths:**
- Walk-forward CV with purge/embargo is implemented in `gbm_predictor`.
- Hot-reload from `models/active/<agent_id>.json` works (`gbm_predictor/main.py`).

**Weaknesses (re-confirming the ML report):**
1. **No settlement/outcome ledger.** Cannot measure hit rate, Brier, calibration, realized return by model, or trade attribution.

4. **News impact signals are not yet routed to risk or sizing.** The schema has no `side` / `quantity`, so the only path is shadow-mode logging. The ML report recommends keeping it that way until calibration is in place — I agree.
5. **The `pairs` agent is a stub.** It's listed in `__init__.py` but no implementation. Cointegration is genuinely hard to get right (lookahead risk on the cointegration test, regime-dependent half-life, borrow-cost asymmetry); leaving it as a stub until cointegration infrastructure is production-ready is correct.

### 1.6 Portfolio math & live portfolio

`libs/fincept-core/src/fincept_core/portfolio.py:apply_fill_to_position` is the canonical function. Pinned by tests in both `services/backtester/tests/test_engine.py` and `services/portfolio/tests/test_state.py` (per the docstring). Good.

`services/portfolio/src/portfolio/store.py:PositionStore` and `state.py` are referenced but not opened in detail. From the schemas + risk snapshot interaction, the store is Redis-backed and indexed by `strategy_id`.

**Gap:** No public analytics tool that takes a `PositionStore` and computes a live Sharpe / live drawdown / live exposure. The operator's view of "how is my paper trading doing" requires either reading a backtest report (which is offline) or eyeballing fills. **A live portfolio analytics tool is the highest-value analytics addition.**

### 1.7 Backtester

`services/backtester/src/backtester/engine.py` is well-designed:
- Per-bar sequencing: `strategy.on_bar` → broker `on_bar` → position update → borrow accrual → equity snapshot. Ordering is correct.
- PIT integrity is real: orders submitted on bar T fill against bar T+1 (the `submitted_in_bar` set).
- Risk gate is optional; when set, it calls the same `risk.check_intent` as live.
- Cost model in `backtester/costs.py` includes maker/taker fees, spread, and borrow accrual on short positions. Realistic.

`backtester/report.py:compute_metrics` returns a `BacktestReport` with:
- `starting_cash`, `final_equity`, `total_return_pct`
- `sharpe`, `max_drawdown_pct`, `longest_drawdown_bars`
- `n_fills`, `fees_paid_total`, `borrow_paid_total`
- `n_rejections`, `rejection_reasons` (prefix-bucketed — nice)
- `per_symbol` stats
- `equity_curve`, `trades`

**Missing from report (sorted by quant value):**
1. **Sortino ratio** — separates downside from upside vol, standard for asymmetric-return strategies.
2. **Calmar ratio** — annualized return / max drawdown. Standard for fund-of-fund style ranking.
3. **Win rate**, **avg win / avg loss ratio**, **profit factor**, **expectancy** — basics of any trade-by-trade evaluation.
4. **Exposure %** — time-weighted fraction of equity deployed. A 5% return with 100% exposure is far less impressive than 5% with 20% exposure.
5. **Baseline comparison** — same strategy on a buy-and-hold of the same symbols, same window. A Sharpe of 1.2 sounds good; Sharpe of 1.2 vs buy-and-hold Sharpe of 1.4 is not.
6. **Recovery time** — bars from the deepest drawdown trough to a new equity high. Important for risk budgeting.
7. **Rolling drawdown series** — operator dashboards want the *path* of the drawdown, not just the max.
8. **Tail metrics** — kurtosis, skew, hit rate on the worst 5% of bars.

**Sharpe formula drift risk:** `analytics.compute_sharpe` uses `np.std(..., ddof=1)` (sample std). `backtester.report._sharpe` uses `var = sum(...)/ (n-1)` (also sample std) but the population of the returns list excludes any `prev_eq <= 0` rows (those are 0.0). These two methods can disagree slightly for the same equity curve if there's a startup cash-drag row.

### 1.8 Paper OMS

`services/oms/src/oms/paper.py:PaperFiller` is clean and injectable:
- Latency modeled as Gaussian noise around 50ms ± 15ms.
- MARKET orders pay half-spread (3 bps default) plus 5 bps taker fee.
- LIMIT orders fill at the limit price with 1 bps maker fee.
- The `rng` and `clock` are constructor-injectable so tests are deterministic.

`oms/processor.py` is referenced but not opened. The flow is `risk.check_intent` → `process_intent` → publish state events + Fill → `fincept_db.audit.append`. This is the right shape.

**Gap:** No paper-fills slippage model keyed to order size relative to volume. A 1000-share market order on a 100-share-ADV stock gets the same price as a 100-share order. Acceptable for paper trading but document the assumption.

### 1.9 Risk gate (`services/risk/src/risk/checks.py`)

The check is pure given a `RiskContext`, which is built by `risk/snapshot.py:build_context` from the live `PositionStore`. Three checks run today:

1. Kill switch (global override).
2. Per-symbol notional cap (`MAX_NOTIONAL_USD_PER_SYMBOL`, default 10 000).
3. Gross notional cap (`MAX_GROSS_NOTIONAL_USD`, default 50 000).

**Critical gap — `MAX_DAILY_LOSS_USD=2000` is configured but never enforced.** Verified by grep:
- Configured in `libs/fincept-core/src/fincept_core/config.py:72`.
- Defined in `services/risk/tests/test_checks.py:59` as a fixture parameter.
- Not referenced anywhere in `services/risk/src/risk/checks.py` or `risk/snapshot.py`.
- Flagged in the worktree audit `docs/project-understanding/07-risks-and-safety-gates.md:244` as a known issue.

**Why this is the #1 thing to fix:** A dad-as-operator paper-trading deployment with a kill switch but no daily-loss cap means a runaway strategy can rack up paper losses all day until the operator notices. The cap is one of the most basic "no, you don't get to keep trading today" controls in retail risk management.

**Other risk-gate gaps:**
- **No per-strategy drawdown cap.** The kill switch is global; a single bad strategy can trip it for everyone. For multi-strategy use, each strategy should have its own drawdown ceiling before tripping the global switch.
- **No correlation-aware sizing.** Two strategies holding 2x gross on the same symbol both pass the per-symbol cap individually but together exceed it... actually no, `notional_by_symbol` aggregates across strategies. So the per-symbol cap is correctly global. Good. **But** two strategies holding similar correlated baskets (e.g., 100% tech) can each pass the gross cap while the portfolio is effectively much more concentrated than the gross number suggests.
- **No "single-order cap".** A 9 999 USD order (under the per-symbol cap) on a 10 000 USD starting account is approved. For a small paper account this is a single-bet-all-in pattern. Consider a `MAX_SINGLE_ORDER_NOTIONAL_USD` cap.
- **No ADV (average daily volume) participation cap.** A paper order for 50% of ADV will get filled at the limit price in the simulator. Document the assumption; consider adding an optional `MAX_PARTICIPATION_PCT` for paper realism.
- **Kill switch persistence** — already noted. For a paper-trading system this is OK; flag in docs.

### 1.10 Reason format

Risk-check `reasons` are free-form strings. `backtester/report.py:_rejection_breakdown` parses them by splitting on the first `:` to get a prefix. Examples from the live code:
- `"kill_switch_engaged"`
- `"no_reference_price:BTC-USD"`
- `"per_symbol_notional_breach:BTC-USD:9000 > 5000"`
- `"gross_notional_breach:55000 > 50000"`

The prefix-based bucketing is fine for the report, but the format is fragile: any future reason that contains a `:` for an unrelated reason would break the breakdown. **Recommendation:** introduce a `RiskReasonCode` enum and a `code: str` + `detail: str` schema; keep the string `reasons` field for backward compat but also expose `codes: list[RiskReasonCode]`.

### 1.11 Prediction log

`libs/fincept-core/src/fincept_core/prediction_log.py`:
- Append-only JSONL, one file per agent under `data/predictions/<agent_id>.jsonl`.
- `read(limit, since_ns, model_name)` supports filter+sort.
- `stats()` returns count, mean_confidence, long/short/flat counts.

**Strengths:**
- Atomic single-write per line (sub-PIPE_BUF on POSIX, sub-cluster on NTFS). The docstring is honest about the limit (no guarantee if the line exceeds 4 KB on POSIX).
- Tolerance for malformed lines — same pattern as `api.promotions`.

**Weaknesses (for an "explainability" goal):**
- No per-prediction rationale.
- No top-k contributing features.
- No regime label (regime_agent emits `RegimeSignal` to a stream, but the prediction log doesn't link to it).
- No link to the source signals (`Decision.source_signals: list[str]` exists, but the prediction row doesn't carry the IDs).

**For the operator view ("why did this signal fire?"), the prediction row is too thin.** Suggested additional optional fields (all backward-compatible because the schema is `frozen, extra="forbid"` — would need a `schema_version=2` row or a sidecar `meta.json` per agent). **Easiest path:** keep the row lean, add a per-agent sidecar `data/predictions/<agent_id>.explain.jsonl` for human-readable explanations. That sidecar can be optional and ignored by older readers.

### 1.12 Calibration

Calibration infrastructure exists in two places:
- `experiments/news-impact-model/src/news_impact_model/evaluation.py:calibration_curve` — bucketed reliability data, used offline in the news-impact experiment.
- No equivalent in `gbm_predictor`. The ML report flagged this as a top-priority gap.

**Recommendation:** create `libs/fincept-core/src/fincept_core/calibration.py` with a `reliability_table(predictions, n_buckets=10)` function. Use it in both places. The `model_dossier` and `meta.json` should store a sidecar `calibration.json` with the bucket counts, ECE, Brier, and log-loss on the validation fold.

### 1.13 News / sentiment

`services/agents/src/agents/sentiment_agent` and `news_alpha_predictor` are gated by LLM API keys (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`) and the calibration policy in `news_alpha_predictor/evaluate.py`. The `news_impact_agent` is the high-level orchestrator. The experiment (`experiments/news-impact-model`) has its own evaluation framework with calibration curves, purged walk-forward, and event-type holdouts.

**Strengths:**
- Schema (`NewsImpactSignal`) carries `confidence`, multi-horizon `q10/q50/q90`, sample size, similar_event_ids, source_urls, and `metadata`. That's good for explainability.
- The signal is a *signal* — no side, no quantity, no venue. The contract keeps risk/OMS as the only execution gates. Correct posture.
- The model is gated behind a candidate evaluation policy.

**Weakness:** No demo end-to-end. The `scripts/test_sentiment_pipeline.py` likely exercises it locally, but in the absence of a stable news provider the live path is "LLM key required, call and hope." Document a fallback: if no LLM key, the sentiment agent should publish a `SentimentSignal` with `confidence=0.0` and skip, not fail.

## 3. Safe Implementation Patch

Four concrete, reversible patches. Each is small enough to land in one PR and each has accompanying tests. All four are paper-only — none touches the live-brokerage gate.

### Patch 1 — Enforce `MAX_DAILY_LOSS_USD` in the risk gate

**Files:** `services/risk/src/risk/checks.py`, `services/risk/src/risk/snapshot.py`, `services/risk/tests/test_checks.py`

The intent: extend `RiskContext` with a `daily_realized_pnl: Decimal` and a `daily_unrealized_pnl_estimate: Decimal` snapshot. `check_intent` adds a check that rejects (or shrinks) any intent whose execution would push combined daily P&L below `-MAX_DAILY_LOSS_USD`. The function stays pure given a richer `RiskContext`.

Sketch (final code should also handle the "approaching the cap" case via `reduced_notional_usd`):

```python
# services/risk/src/risk/checks.py — add to RiskContext
@dataclass(frozen=True)
class RiskContext:
    notional_by_symbol: dict[str, Decimal] = field(default_factory=dict)
    gross_notional: Decimal = Decimal(0)
    kill_switch_engaged: bool = False
    # NEW: today's P&L snapshot (realized so far + estimate of unrealized).
    daily_realized_pnl: Decimal = Decimal(0)
    daily_unrealized_pnl_estimate: Decimal = Decimal(0)
    # NEW: wall-clock of the "day" boundary; a per-process configurable
    # at the OMS layer (default 00:00 America/Chicago to match the
    # dashboard's day boundary).
    trading_day_started_at_ns: int = 0
```

```python
# services/risk/src/risk/checks.py — append after the gross-notional check
combined_daily_pnl = ctx.daily_realized_pnl + ctx.daily_unrealized_pnl_estimate
if combined_daily_pnl < -Decimal(settings.MAX_DAILY_LOSS_USD):
    reasons.append(
        f"daily_loss_breach:{combined_daily_pnl} < -{settings.MAX_DAILY_LOSS_USD}"
    )
```

# Sortino — same annualization as Sharpe, but denominator uses downside
# deviation only (returns < 0).
def _sortino(returns, *, bars_per_year, target=0.0):
    if len(returns) < 2:
        return None
    excess = [r - target for r in returns]
    downside = [r for r in excess if r < 0]
    if not downside:
        return None
    dd = math.sqrt(sum(r * r for r in downside) / len(returns))  # full-population denom
    if dd == 0:
        return None
    return (sum(returns) / len(returns)) * bars_per_year / (dd * math.sqrt(bars_per_year))

## 5. Assumptions & Data Limitations

### 5.1 Assumptions

- The audit assumes the project is paper-trading-first and stays that way through the next two milestones. **No live-trading assumption is made.** The Alpaca adapter exists but is treated as "paper Alpaca" (the default `ALPACA_BASE_URL=https://paper-api.alpaca.markets`).
- The audit assumes `services/api` and the dashboard are out of scope for the next two weeks of work. Quant/ML patches here should land without requiring dashboard changes; the existing API surface is rich enough to expose new endpoints.
- The audit assumes the kill switch is acceptable as in-memory state for a paper deployment, **as long as** the operator is informed via a dashboard banner when the OMS restarts (a small follow-up).
- The audit assumes the existing `PredictionLog` JSONL convention stays. Sidecar files for explainability and calibration are consistent with that convention.
- The audit assumes the strategy host can pick up new `StrategyConfig` fields without breaking currently-running strategies. New optional fields with `None` defaults are safe by construction.
- I did not open every file in the 12 services — the audit's depth is proportional to how much the surface is touched by quant/ML work. Files NOT opened: `services/ingestor/*`, `services/jobs/*`, `services/api/*`, `services/features/*`, `services/strategy_host/runner.py`, `services/agents/src/agents/gbm_predictor/*` (training specifics), `services/agents/src/agents/sentiment_agent/*` (LLM router), `services/agents/src/agents/regime_agent/*` (FRED rules).

### 5.2 Data limitations

- **No real paper-trading history yet.** The `paper_spine_replay.py` is a deterministic fakeredis fixture. We do not have days/weeks of "this strategy ran against this universe" data. Any production-grade metric (Sortino, Calmar, profit factor) needs a longer history to be meaningful; until then, the metric is computed but its values are illustrative, not statistical.
- **No service-backed Timescale test run.** The `data.get_bars` tool requires Timescale; the test suite marks Timescale-backed tests as `skip` when the service isn't running. So the analytics tools are not exercised end-to-end against real bars in CI. The fakeredis path is proven; the Timescale path is not.
- **No news provider integration live.** Both `sentiment_agent` and `news_impact_agent` require external API keys that were not present in the audited environment. The schemas are correct; the LLM-router behavior is not validated against a live provider in this pass.
- **No OpenBB backend available.** The `analytics.compute_*` tools read from `fincept_db.bars:read_bars`, which is the local DB — not OpenBB. OpenBB integration is in `libs/fincept-tools/src/fincept_tools/research/openbb.py` and is gated by `OPENBB_API_URL`. If OpenBB is the data source, the analytics tool path is the same but the underlying bars are different.
- **The `pairs` agent is a stub.** Cointegration-based pairs trading cannot be audited; assume the standard "stale cointegration, look-ahead, regime-shift" risks apply if/when it lands.
- **The `alpaca.live` strategy config uses class `position_tracker` with 11 symbols.** It's a "track only" strategy (model_binding=null). It will not emit orders. Operators looking for "what is strategy X actually doing" should not be confused by it appearing under "running strategies."
